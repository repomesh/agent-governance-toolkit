# Adapter mediation

ACS adapters mediate governed framework activity by placing the ACS runtime on the only supported call path for the wrapped operation. A model adapter evaluates `pre_model_call` before invoking the model and evaluates `post_model_call` before returning the model response. A tool adapter evaluates `pre_tool_call` before invoking the tool and evaluates `post_tool_call` before returning the tool result. Agent run adapters use `input` before the agent body and `output` before the final return.

A DENY verdict at a pre point prevents the wrapped model, tool, runner, filter continuation, or upstream app from executing. An ALLOW verdict executes the wrapped callable with the effective policy target. If the pre verdict carries an allowed transform then the wrapped callable receives the transformed request or arguments. If the post verdict carries an allowed transform then the adapter returns the transformed response or result.

The mediation guarantee is a wrapper guarantee. It holds when the host publishes and calls only the guarded object, guarded function, middleware, hook, provider, or filter. The host must not retain or publish a separate reference to the original unwrapped model or tool. The host must not call alternate framework methods that are outside the adapter surface. Adapters that can identify unsafe alternate methods block those methods. Adapter surfaces that receive a `next` delegate or explicit upstream callable depend on the framework contract that the framework invokes the adapter in order.

The Rust Rig adapter stores its inner tool behind the `GuardedRigTool` value and exposes only the `ToolDyn` implementation from that value. A host that keeps a clone of the original `Arc<dyn ToolDyn>` can still call it directly, so Rig integrators must register and retain only the guarded tool. Use `GuardedRigTool::with_ambient_snapshot` to provide host context such as `snapshot.ifc.source_labels` to both `pre_tool_call` and `post_tool_call`. Use `GuardedRigTool::call_with_result` before erasing the tool behind `dyn ToolDyn` when the host needs post-tool ACS metadata such as `result_labels`. The .NET delegating chat client and MCP provider keep the wrapped client or callable in private fields. The .NET Semantic Kernel and AutoGen middleware receive the framework continuation per invocation, so the host contract is that the framework registers the ACS middleware in the invocation pipeline.

The Python and Node duck typed adapters return proxy objects or wrapped functions. Those runtimes cannot prevent a host that saved the original object from calling it directly. Integrators must discard unwrapped references after registration and must register the guarded proxy with the framework. The GHCP integration is hook based. It mediates the permission hook values and relies on GHCP to honor deny decisions and modified arguments before the underlying tool call proceeds.

Current adapter inventory includes the Rust Rig tool adapter in `integrations/rig`. Python includes LangChain runnable, OpenAI client, OpenAI Agents runner, Anthropic client, MCP tool and server, Semantic Kernel function and filter, AutoGen agent, CrewAI crew, and LiteLLM proxy adapters. Node includes generic model middleware and model wrapping, LangChain runnable, OpenAI Agents runner, Anthropic client and tool, MCP tool provider, OpenClaw explicit hooks, and GHCP hooks. .NET includes MCP tool provider, Semantic Kernel function filter, AutoGen middleware, delegating chat client, generic agent middleware, and generic tool invocation filter.

## Failure modes

ACS distinguishes mediation failures from downstream callable failures. A mediation failure is any failure inside the guarded path that ACS owns, including a policy dispatcher that raises, a verdict that fails to deserialize, an annotation dispatch that fails, an approval resolver that throws, returns null, or returns an unrecognized outcome, or an adapter receiving an unsupported framework method. Every mediation failure fails closed and produces a DENY with a `runtime_error:*` reason such as `runtime_error:approval_resolver_failed`, `runtime_error:adapter_unsupported`, or `runtime_error:annotation_failed`. Pre-call mediation failures and unsupported method failures happen before the wrapped model or tool runs. Post-call mediation failures, including post-model approval failures after buffered streaming, happen after upstream returns and block the guarded response from reaching the caller.

A downstream callable failure is an exception raised by the wrapped model or tool itself after `pre_tool_call` or `pre_model_call` has allowed the invocation. ACS does not convert that exception into a policy denial. The exception propagates verbatim to the caller and the corresponding post point does not run, because the failure originates in host code that ACS was asked to invoke rather than in the mediation that ACS performs. Hosts that need to govern a failed tool result must surface a structured result to the post point instead of raising.

The `tool_call_id` is optional and handled uniformly across every SDK host API as specified in the snapshot and policy-input contract under tool call identity.
